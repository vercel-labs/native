# Embedded App

`EmbeddedApp` drives the runtime without the full platform event loop. Use it for embedding zero-native in an existing application, game engine, or custom render loop.

## Usage

```zig
var embedded = zero_native.embed.EmbeddedApp.init(my_app.app(), my_platform);

try embedded.start();

// In your render loop:
try embedded.frame();

// On resize:
try embedded.resize(new_surface);

// On shutdown:
try embedded.stop();
```

## How it works

`EmbeddedApp` wraps a `Runtime` and an `App`. Each method dispatches a platform event via `runtime.dispatchPlatformEvent`, giving you full control over the event loop while still using zero-native's runtime, bridge, and window management.

## Mobile examples

The repository includes full mobile host examples:

- `examples/ios` - Xcode project with a Swift `UIViewController`, `WKWebView`, and `zero_native.h` bridge.
- `examples/android` - Gradle/Kotlin project with JNI and CMake wiring for `libzero-native.a`.

Both examples expect a local `libzero-native.a` built from the repository and copied into the path documented in each example README.

## Mobile host contract

Mobile embedding uses the same runtime lifecycle through a thin native host:

<table>
  <thead>
    <tr>
      <th>Host event</th>
      <th>zero-native call</th>
      <th>Current examples</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>View/controller creation</td>
      <td><code>zero_native_app_create</code>, then <code>zero_native_app_start</code></td>
      <td>Swift <code>viewDidLoad</code>; Android <code>onCreate</code></td>
    </tr>
    <tr>
      <td>Content size or surface changes</td>
      <td><code>zero_native_app_resize</code>, then <code>zero_native_app_frame</code></td>
      <td>Swift <code>viewDidLayoutSubviews</code>; Android <code>surfaceChanged</code></td>
    </tr>
    <tr>
      <td>Touch input</td>
      <td><code>zero_native_app_touch</code>, then <code>zero_native_app_frame</code></td>
      <td>Android forwards <code>MotionEvent</code> data; iOS keeps touch handling in UIKit/WebKit for now</td>
    </tr>
    <tr>
      <td>Shutdown</td>
      <td><code>zero_native_app_stop</code>, then <code>zero_native_app_destroy</code></td>
      <td>Swift controller deinit; Android <code>onDestroy</code></td>
    </tr>
  </tbody>
</table>

The mobile examples keep navigation/header UI in UIKit or Android views and keep rich content in `WKWebView` or Android `WebView`. Safe areas, orientation, keyboard avoidance, back gestures, and platform lifecycle ownership stay in the native host. The generic desktop `ShellView` tree is not mapped to UIKit or Android `ViewGroup` yet, so mobile app metadata is currently descriptive and packaging-oriented rather than a runtime materialized view tree.

## Testing with EmbeddedApp

```zig
var null_platform = zero_native.NullPlatform.init(.{});
var state: u8 = 0;
var embedded = zero_native.embed.EmbeddedApp.init(.{
    .context = &state,
    .name = "embedded",
    .source = zero_native.WebViewSource.html("<p>Embedded</p>"),
}, null_platform.platform());

try embedded.start();
// null_platform.loaded_source now contains the loaded HTML
```
