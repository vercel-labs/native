# Built-in Components

zero-native includes a native-rendered component catalog with shadcn-style defaults. These components are owned by zero-native, rendered through the retained canvas/GPU surface, and styled by design tokens instead of platform widget skins.

The default visual language follows shadcn: neutral surfaces, sharp typography, subtle borders, focused states, compact density, and token-driven color, radius, shadow, blur, and motion. Geist is the default sans family and Geist Mono is the default mono family. Applications can override tokens globally or per widget when they need their own product language.

In markup views these render through their `.zml` elements (`<button>`, `<checkbox>`, `<slider>`, `<card>`, ... — see [Native UI](/native-ui) for the element table); the full catalog below is available programmatically through the `canvas.Ui` builder and `builtinComponentWidget`. Use these components for app UI that should be fast, composable, and visually consistent across platforms. Use WebViews when a feature is naturally web-shaped, such as a browser surface, document editor, rich preview, embedded web app, or third-party content.

## Runtime Contract

The canonical catalog lives in `zero_native.canvas.builtin_component_kinds` and `zero_native.canvas.builtin_component_names`. Each descriptor reports its shadcn style, root widget kind, semantic role, and whether the component is composite.

Use `zero_native.canvas.builtinComponentWidget(...)` to create a default component foundation, then pass explicit children, layout, state, semantics, commands, or token overrides as needed. Composite components are still zero-native components; they assemble retained primitives under the same token and accessibility model.
