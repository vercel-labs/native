# Quick Start

Native SDK is the complete toolkit for building native desktop and mobile applications. An app is a declarative markup view plus Zig logic, rendered by Native SDK's own engine into a real OS window — no browser in the binary, no npm in the loop.

## Prerequisites

- [Zig 0.16.0+](https://ziglang.org/download/)
- macOS 11 or newer, Linux, or Windows
- Node.js with npm (only for installing the CLI, and for apps that [embed web content](/frontend))

Install the `native` CLI:

```bash
npm install -g @native-sdk/cli
```

## Create an app

```bash
native init my_app
cd my_app
```

## Run it

```bash
zig build run
```

A native window opens with a working counter. The whole view is `src/app.zml`:

```html
<column gap="12" padding="16">
  <row gap="8" cross="center">
    <text grow="1">Counter</text>
    <button size="sm" variant="ghost" on-press="reset">Reset</button>
  </row>
  <row gap="8" main="center" cross="center" grow="1">
    <button variant="secondary" on-press="decrement">-</button>
    <text>{count}</text>
    <button variant="primary" on-press="increment">+</button>
  </row>
  <status-bar>count: {count}</status-bar>
</column>
```

The markup binds values (`{count}`) and dispatches messages (`on-press="increment"`); it can never mutate state. All logic lives in `src/main.zig`:

```zig
pub const Msg = union(enum) {
    increment,
    decrement,
    reset,
};

pub const Model = struct {
    count: i64 = 0,
};

pub fn update(model: *Model, msg: Msg) void {
    switch (msg) {
        .increment => model.count += 1,
        .decrement => model.count -= 1,
        .reset => model.count = 0,
    }
}
```

That is the whole loop: the model holds state, messages describe what happened, `update` is the only place state changes, and the view re-derives from the model. [App Model](/app-model) walks through how the pieces connect.

## Edit while it runs

`src/app.zml` is embedded into the binary and watched during development. Edit it while the app runs — change a color token, add a button — and the window updates within about two seconds without losing the count. Parse failures keep the last good view on screen.

Validate markup without building:

```bash
native markup check src/app.zml
```

Errors come back with `file:line:column` and a teaching message. `native markup lsp` provides the same diagnostics plus completion and hover in your editor.

## Test it

```bash
zig build test
```

The generated `src/tests.zig` builds the real view from the markup, finds the `+` button, dispatches its press exactly like the runtime would, and asserts the model and the rebuilt view — headless, on any machine.

## Drive it from the outside

Build with automation enabled and the running app publishes an accessibility snapshot any script (or AI agent) can assert on:

```bash
zig build -Dautomation=true
./zig-out/bin/my_app &
native automate wait
native automate assert 'role=button name="Reset"' 'count: 0'
```

The generated CI workflow runs exactly this flow headless under Xvfb on Linux — see [Testing in CI](/testing/ci).

## Style with design tokens

Color and radius attributes reference design tokens by name — `background="surface"`, `foreground="text_muted"`, `radius="md"` — validated at check/compile time and re-resolved live when the theme changes, so dark mode works without app code. See [Native UI](/native-ui#styling-with-design-tokens).

## Package it

```bash
zig build package
native doctor --manifest app.zon --strict
```

See [Packaging](/packaging) for platform targets and [Code Signing](/packaging/signing) for distribution.

## Apps with web content

Native SDK can also embed web frontends — an existing React, Next.js, Svelte, or Vue app running in a WebView inside the native shell, with a guarded bridge to native code:

```bash
native init my_app --frontend next
```

Frontend options: `next`, `vite`, `react`, `svelte`, `vue`. This is a secondary path for teams bringing existing web UI along; see [Web Content](/frontend) for the full workflow.
