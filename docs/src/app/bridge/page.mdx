# Bridge

The bridge connects JavaScript in the WebView to native Zig handlers via JSON messages.

## Architecture

```
WebView JS                       Zig Runtime
──────────                       ───────────
window.zero.invoke(cmd, payload)
        │                              │
        ├──── JSON message ───────────►│
        │                         Size check (1 MiB max)
        │                         Policy check (origin + permissions)
        │                         Handler lookup + execute
        │◄─── JSON response ──────────┤
```

## Defining a handler

```zig
fn ping(context: *anyopaque, invocation: zero_native.bridge.Invocation, output: []u8) anyerror![]const u8 {
    _ = invocation;
    const self: *App = @ptrCast(@alignCast(context));
    self.ping_count += 1;
    return std.fmt.bufPrint(output, "{{\"message\":\"pong\",\"count\":{d}}}", .{self.ping_count});
}
```

The handler writes its JSON result into the provided `output` buffer (max 1 MiB) and returns a slice of it. Results must be valid JSON values; invalid raw text is rejected with `handler_failed`. When returning user data as a string, use the bridge helper so quotes and control characters are escaped:

```zig
return zero_native.bridge.writeJsonStringValue(output, user_supplied_name);
```

## Wiring the dispatcher

```zig
fn bridge(self: *App) zero_native.BridgeDispatcher {
    self.handlers = .{.{ .name = "native.ping", .context = self, .invoke_fn = ping }};
    return .{
        .policy = .{ .enabled = true, .commands = &policies },
        .registry = .{ .handlers = &self.handlers },
    };
}
```

## Calling from JavaScript

```javascript
const result = await window.zero.invoke("native.ping", { source: "webview" });
console.log(result); // { message: "pong from Zig", count: 1 }
```

## Large data resources

The JSON bridge is the control plane. For larger byte payloads, return a resource descriptor from Zig and let the frontend consume it through browser-native `fetch()` APIs:

```json
{
  "kind": "resource",
  "id": "opaque-id",
  "url": "zero://native/resource/opaque-id",
  "mime": "application/octet-stream",
  "size": 1048576
}
```

`size` is omitted when the native source cannot know the length up front.

```javascript
const resource = await window.zero.invoke("native.export", {});
if (!window.zero.resources.isDescriptor(resource)) {
  throw new Error("native.export did not return a resource");
}
const response = await window.zero.resources.fetch(resource);
const bytes = new Uint8Array(await response.arrayBuffer());

const stream = await window.zero.resources.stream(resource);
const reader = stream.getReader();
```

Async Zig handlers can use `AsyncResponder.resourceBytes` to register bytes with the runtime and complete the command with a small descriptor instead of embedding the bytes in the JSON response:

```zig
fn exportBytes(context: *anyopaque, invocation: zero_native.bridge.Invocation, responder: zero_native.bridge.AsyncResponder) anyerror!void {
    _ = context;
    _ = invocation;
    try responder.resourceBytes("large payload", zero_native.BridgeResourceOptions.download("payload.txt", "text/plain"));
}
```

For true source streaming, use `AsyncResponder.resourceStream` with a provider. The provider is pulled by the native resource handler in chunks, so the full payload does not need to be copied into the runtime or native host before the browser starts consuming it:

```zig
const ExportStream = struct {
    file: std.fs.File,

    fn read(context: *anyopaque, output: []u8) anyerror!usize {
        const self: *@This() = @ptrCast(@alignCast(context));
        return self.file.read(output);
    }

    fn close(context: *anyopaque, reason: zero_native.BridgeResourceCloseReason) void {
        _ = reason;
        const self: *@This() = @ptrCast(@alignCast(context));
        self.file.close();
    }
};

try responder.resourceStream(.{
    .context = &export_stream,
    .read_fn = ExportStream.read,
    .close_fn = ExportStream.close,
    .size = known_size_or_null,
}, zero_native.BridgeResourceOptions.download("export.bin", "application/octet-stream"));
```

Resource responses also need a registry attached to the runtime:

```zig
var bridge_resources = zero_native.BridgeResourceRegistry.init(allocator);
defer bridge_resources.deinit();

var runtime = zero_native.Runtime.init(.{
    .platform = platform.platform(),
    .bridge = app.bridge(),
    .bridge_resources = &bridge_resources,
});
```

Resources are one-shot by default and expire after `BridgeResourceDefaultTtlNs` so abandoned descriptors do not live forever. The native WebView resource stores enforce the same TTL for resources that are registered but never fetched. Reusable resources are an advanced opt-in:

```zig
try responder.resourceBytes(
    bytes,
    zero_native.BridgeResourceOptions.download("payload.bin", "application/octet-stream").reusable().withoutTtl(),
);
```

The byte resource path currently works with the system WebView engines on macOS (WKWebView) and Linux (WebKitGTK). `resourceStream` registers provider callbacks so those hosts pull chunks from Zig as the native resource handler serves the browser response. Hosts that do not yet have a native resource interceptor reject registration through the normal bridge error path with `handler_failed`.

`zero.resources.stream()` exposes the browser response stream. For `resourceStream` descriptors, the native host pulls from the Zig provider in chunks; for `resourceBytes` descriptors, the source is still a buffered byte slice.

### Benchmarking large transfers

When measuring bridge throughput, compare at least three payload sizes: 1 MiB, 16 MiB, and 128 MiB. The recommended baseline is:

- JSON bridge with plain JSON data for small structured results.
- JSON bridge with base64 for binary data.
- Resource descriptor plus `zero.resources.fetch(...).arrayBuffer()` for binary data.

The resource path should avoid the base64 size overhead and keeps large bytes out of `evaluateJavaScript`/`ExecuteJavaScript` response strings.

## Invocation

When a handler is called, it receives an `Invocation` with:

- `request.id` -- caller-provided request ID (max 64 bytes)
- `request.command` -- command name (max 128 bytes, no `/` or spaces)
- `request.payload` -- JSON payload string
- `source.origin` -- origin of the requesting page (e.g. `zero://app`)
- `source.window_id` -- which window sent the request
