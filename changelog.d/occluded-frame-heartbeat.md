fix: **Occluded windows no longer spin the frame clock at full rate**: when a macOS window is fully occluded (covered, minimized, hidden app), logical frame completions now throttle to a ~1 Hz heartbeat instead of the display grid — an armed frame loop (a playing scrubber, a running tween) stays gently current instead of rebuilding its display list 60+ times a second for glass nobody can see, which pinned a core and tripped the OS CPU-usage watchdog. De-occlusion supersedes the parked heartbeat emission immediately, so full cadence and the retained-canvas glass flush return without dropping a beat, and the first-present exemption is untouched. The Windows host applies the same heartbeat while the top-level window is minimized (the one occlusion signal Win32 reports reliably for its paint path); GTK deliberately keeps full cadence — no cross-backend visibility signal is dependable enough to throttle on — with the reasoning documented at its scheduler. Idle apps still emit zero frames.
- **Occluded completions are no longer latency endpoints**: frame events now carry an `occluded` flag, and the runtime resolves a pending input against an occluded logical completion without stamping an input latency — the completion's timestamp is deliberate heartbeat pacing, not a present, and stamping it manufactured budget overruns (a covered window's click could read as 150+ ms against an 8.3 ms budget). Real presents, including the de-occlusion flush, still measure exactly as before.
