fix: **The Chromium (CEF) host builds and runs again, and the gate now compiles it**: Chromium-engine builds failed on current macOS SDKs with a wall of "pointer is missing a nullability type specifier" errors from system headers plus `<cstring>`/`<math.h>` breakage — the build flags passed the SDK's `usr/include` as a plain `-I`, which both shadowed libc++'s C-header wrappers in Objective-C++ and demoted SDK headers to user headers (unleashing their nullability-completeness noise). The flag is now `-isystem`, searched after the toolchain's libc++/libc headers, in every example build, the packaged-app build, and the `native create` template. With the host finally compiling, the accumulated never-compiled drift surfaced and is fixed: the per-window child WebView exports had C++ linkage (they are now declared in `appkit_host.h`, the C ABI contract, so both hosts keep C linkage), app timers and the cross-thread wake are implemented for real (the runtime scheduler needs them on every engine), image decode reuses the same ImageIO decoder as the AppKit host, and the gpu-surface/native-view surface reports unsupported explicitly instead of not linking. `zig build test-webview-cef-link` now works with the default `third_party/cef/macos` layout (the root build rebases the CEF dir for example sub-builds), and `scripts/gate.sh` runs it whenever `src/platform/macos/` changes (fast tier; always in full) so a cef_host.mm edit can never again land without seeing a compiler — it skips with a loud priming warning when no CEF layout is installed. Verified live: the WebView example boots under Chromium, `native.ping` answers over the bridge, and child WebViews create/resize/navigate/close against a real page.
