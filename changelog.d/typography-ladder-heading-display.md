feature: **Display-scale text through the token ladder**: two new typography token steps — `heading_size` (28) and `display_size` (48) on `TypographyTokens`, themable through `TypographyTokenOverrides` like every token — reachable from markup as `<text size="heading">`/`<text size="display">` and from Zig views as `ElementOptions.size = .heading/.display`, so a dashboard hero stat, a timer numeral, or a pricing figure is finally expressible declaratively instead of via span scales. The rungs replace the body base for text leaves only (line height, wrapping, wrapped-height reservation, and the layout audit all follow), do not density-scale (density scales chrome, never glyphs), and never change semantics — a display-size stat announces exactly like a body-size one.
- **Two axes, kept apart with teaching errors**: `size` on controls stays the control scale (`default|sm|lg|icon`); `heading`/`display` on a non-text element is a validation/compile error naming text as their home, unknown size values teach the whole vocabulary, and numeric sizes stay refused by design — retheme the typography tokens to move the whole scale. Validator, interpreter, and comptime compiler agree (parity pinned); Zig views get a Debug warning for the same misuse.
- **Shipped adoptions**: the system-monitor stat tiles (bold heading-rung values instead of a raw span scale), the calculator's result line (the display rung themed to 36px for the 12-digit mono entry window — one token move instead of a magic 2.6x), and the habits header title (`size="heading"` in markup).
