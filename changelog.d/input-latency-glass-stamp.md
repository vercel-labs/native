fix: **`gpu_input_latency` measures input to glass, not input to the pacing channel**: the metric used to stamp at the next paced frame-completion event, which lands up to a display interval after (or before) the responding present — overstating a fast response by most of a frame and understating a slow one. It now stamps at the RESPONDING present's completion: the first successful present after input receipt records `now − input_timestamp` in the same wall-clock domain every host and automation input timestamp uses. Inputs that present nothing keep the completion-event fallback, which moved after the frame's app dispatch so a present inside that dispatch gets first claim; the frame event that resolves a pending input therefore carries the previous latency and the resolved value rides the next one. The honest number legitimately spans the paced render wait (a couple of display intervals while animations hold the paced loop), so the gpu-components smoke asserts an explicit input-to-glass bound instead of the one-interval budget flag the old stamp happened to satisfy.
