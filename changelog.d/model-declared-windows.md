feature: **Secondary windows are model-declared**: `UiApp.Options.windows_fn` returns the window descriptors that should exist right now (label, canvas label, title, size, resizable, titlebar, `on_close` Msg — presence IS visibility, since the platform window channel is create/focus/close with no hide) and `Options.window_view` builds each declared window's canvas tree by window label. The runtime reconciles the declared set against live windows after every rebuild — creating source-less native windows wearing one gpu_surface view that inherits the main canvas's gpu options, closing what the model stopped declaring — and every dispatched Msg rebuilds every open window's view from the same model, so a settings window's theme picker restyles the main window on the same dispatch. Input from any window dispatches Msgs through that window's own tree with its window identity; a USER close dispatches the descriptor's `on_close` Msg through the new `window_closed` runtime event (the dismissal precedent: the window is already gone as the optimistic echo, the model's next declared set is truth — keep declaring it and it comes back), while model-initiated closes never echo. Closed windows release their label for reopen, canvas-only windows never grow an implicit webview (`createSourcelessShellWindow`), automation snapshots enumerate every window and the widget verbs + `screenshot` address any window's canvas by view label, and window descriptors accept `titlebar = .hidden_inset` (content under a transparent titlebar; macOS `NSWindowStyleMaskFullSizeContentView` + hidden title, traffic lights intact — drag regions and inset-aware headers are a follow-up channel). Budget: `canvas_limits.max_ui_app_windows` (4), loud on excess. The system-monitor example grew a gear chip whose Msg opens a model-declared settings window (theme + sampling), closed by Msg or by the user, driven end to end by automation in its suite.
