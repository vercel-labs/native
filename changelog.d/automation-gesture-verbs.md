feature: **Press-and-hold and right-click are automation-drivable**: `native automate widget-hold <view> <id>` runs the full hold gesture through the real dispatch path — pointer-down at the control's aim point (arming the reserved hold timer exactly as a live press does), the timer fired as the platform would fire it at ~350 ms, then the release the fired hold suppresses — so `on_hold` Msgs are live-drivable instead of tests-only; a target without `on_hold` degrades to the click a real long press is. `widget-context-press <view> <id>` is the secondary-click leg: the full button-1 stream, so the runtime presents the widget's declared context menu (or the editable/selected-text defaults) and otherwise dispatches the hold Msg immediately, the desktop press-and-hold alternative. The hold timer id moved to the platform layer (`platform.press_hold_timer_id`) so the automation dispatch fires the same timer the gesture arms; stale targets fail loudly like widget-click, and the automation protocol version bumped so skewed CLI/app pairs refuse instead of misreading.
