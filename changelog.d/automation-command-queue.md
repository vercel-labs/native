fix: **Rapid-fire automation commands can no longer be lost**: the dropbox's single-entry `command.txt` slot could be overwritten by a back-to-back `native automate` invocation before the app drained it, silently dropping a command (field-confirmed: one of three rapid `menu-command`s vanished). Commands now queue as `command-<n>.txt` entries — each writer claims the next sequence with an exclusive create (a name race is a rescan-and-retry, never an overwrite), the app consumes one entry per presented frame in strict arrival order, and deleting the entry is the consumption ack the CLI waits for before printing `delivered`. The queue is bounded (8 entries); a writer finding it full retries at the consumption cadence up to its existing ~10s budget, then refuses loudly naming the depth. Consumption stays on the `frame_requested` boundary, so one-command-per-frame and session record/replay determinism are unchanged, and the arrival watcher stays quiet on an idle dropbox. Protocol version bumped 5 -> 6: a stale CLI or app on the old single-slot shape is refused up front by the snapshot handshake, and any writer that slips past it (no snapshot yet) times out loudly instead of misbehaving silently.
