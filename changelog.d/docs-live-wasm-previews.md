feature: **Live docs component previews — the engine running in-page via wasm**: `zig build docs-wasm-preview` compiles the retained canvas runtime plus the exact scene catalog the static webp previews render (now shared in `tools/docs_preview_scenes.zig`, so the two pipelines cannot drift) to a self-contained `wasm32-freestanding` module (~828 KB raw, ~306 KB gzip, ~258 KB brotli, zero imports) with an embed-shaped ABI: create-by-scene-name, pointer/key/text/scroll entry points, theme re-skin, a frame tick for engine animations, and a render call that reports whether the retained display list actually changed. Every component page's preview tile upgrades on hover/click/keyboard focus from the instant SSR webp pair to a live engine instance on a canvas that follows the site theme (the light/dark image pair collapses to one canvas) — switches toggle, text fields edit, tabs select, sliders drag, scroll regions wheel-scroll with momentum, and Tab drives the engine's roving widget focus (Escape returns to the page). Honest costs kept honest: the wasm loads only when a preview tile approaches the viewport, the rAF loop parks completely when nothing repaints (zero rAF calls idle), at most four live instances stay resident (older tiles fall back to their static images), and with JS disabled the pages are byte-for-byte the static previews.
- Freestanding target support cost three comptime guards, all in the "honest degradation" pattern the wasi clocks already used: `runtime/clock.zig` and the automation snapshot pid read report 0, and the window-state store + automation dropbox server no-op where no filesystem exists.
- Regenerating the static previews also refreshed three stale pairs (markdown, stepper, timeline) whose committed webps predated the current monospace glyph metrics.
