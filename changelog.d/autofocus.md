feature: **Source-driven autofocus**: keyboard-first flows finally have an app-driven focus channel — `ElementOptions.autofocus` / markup `autofocus="true|{binding}"` moves keyboard focus to a focusable control on the rebuild where it MOUNTS or where the flag turns on. Edge-triggered like the source-wins control reconcile: holding it true never re-steals focus from the user, so declaring it in the view stays honest TEA (focus is a function of the model — notes mounts its inline editor with `autofocus` on note-create, the calculator marks its input focused from the first frame, no clicks). The runtime tracks the previous rebuild's flags per view (16-slot budget, documented in canvas_limits) and applies the same focus write every other focus source performs (view focus + focused/visible ids + invalidation). Markup scopes it to focusable controls with a teaching error on layout/decoration elements, enforced by the validator, both engines, and documented in the LSP/skill/docs. Tested through the real dispatch path: a menu command mounts the editor, typing lands in the model with no click, and a later user click is never stolen back. The effects-channel alternative (`fx.focus(key)`) was deliberately skipped: it would need a new runtime binding and global-key bookkeeping for something the view already knows.
